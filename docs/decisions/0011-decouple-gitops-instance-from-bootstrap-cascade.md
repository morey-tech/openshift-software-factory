# Decouple openshift-gitops-instance from the Bootstrap Cascade to Fix Namespace Termination

## Context and Problem Statement

Running `ansible/teardown.yaml` left the `openshift-gitops` namespace permanently stuck in `Terminating`. Investigation with `oc get namespace openshift-gitops -o jsonpath='{.status.conditions}'` showed:

```
Some content in the namespace has finalizers remaining:
  resources-finalizer.argocd.argoproj.io in 5 resource instances
```

The root cause is a cascade deadlock. The `openshift-gitops-instance` Application (generated by the operands AppSet) manages the ArgoCD CR. It sits in the same cascade chain as every other Application under the `bootstrap` root Application. When `bootstrap` is deleted:

1. `bootstrap` finalizer fires â†’ deletes ApplicationSets
2. ApplicationSets deleted â†’ all child Applications deleted (cascade)
3. `openshift-gitops-instance` finalizer fires â†’ ArgoCD CR deleted â†’ GitOps operator tears down ArgoCD pods
4. **ArgoCD controller is now dead**
5. The 4 remaining Applications (`dev-spaces-instance`, etc.) each have `resources-finalizer.argocd.argoproj.io` â€” but the controller that processes this finalizer is gone
6. Those Applications block `openshift-gitops` namespace termination indefinitely

## Decision Drivers

* Namespace termination must complete cleanly after teardown
* The teardown playbook should not need per-component knowledge to work around the deadlock
* The ArgoCD instance should still be GitOps-managed (auto-synced from the repo)

## Considered Options

* Force-patch finalizers off the stuck Applications during teardown (fragile, requires per-Application knowledge)
* Delete ApplicationSets and wait for child Applications to be cleaned before deleting the namespace (complex, tightly coupled)
* Take `openshift-gitops-instance` out of the operands AppSet cascade chain entirely

## Decision Outcome

Chosen option: decouple `openshift-gitops-instance` from the operands AppSet cascade.

A `"standalone": "true"` field is added to `components/openshift-gitops/instance/config.json`. The operands AppSet selector excludes `standalone=true` entries, so no `openshift-gitops-instance` Application is generated. Instead, the bootstrap playbook applies a standalone `openshift-gitops-instance` Application directly (analogous to how it applies the `bootstrap` root Application).

**`finalizers: []` is required on the standalone Application.** If the cascade finalizer were present, deleting the Application during teardown would trigger ArgoCD to cascade-delete the ArgoCD CR, killing the controller â€” reproducing the exact same deadlock in a different form.

Teardown order after this change:
1. Delete `bootstrap` â†’ wait for full cascade (ArgoCD controller alive throughout; `openshift-gitops-instance` is not in this chain)
2. Delete `openshift-gitops-instance` Application â†’ instant (no finalizer)
3. Delete CSVs in `openshift-operators`
4. Delete `openshift-gitops` namespace â†’ **wait** (GitOps operator in `openshift-gitops-operator` is still running; it processes the ArgoCD CR teardown finalizer cleanly)
5. Delete `openshift-gitops-operator` and `openshift-devspaces` namespaces â†’ wait

### Consequences

* Good, because namespace termination completes cleanly â€” no stuck finalizers
* Good, because the teardown playbook requires no per-component knowledge to handle the deadlock
* Good, because the ArgoCD instance remains GitOps-managed via the standalone Application
* Bad, because `openshift-gitops/instance/config.json` needs a `standalone` field that other `instance/` configs do not have â€” new contributors must understand this convention

## Pros and Cons of the Options

### Force-patch finalizers during teardown

* Good, because no structural changes to AppSets or config.json
* Bad, because teardown must enumerate all stuck Applications â€” tightly coupled to component manifests
* Bad, because it papers over the root cause rather than fixing it

### Standalone Application for openshift-gitops-instance

* Good, because structurally correct â€” the controller that manages ArgoCD cannot be in the same cascade as the resources ArgoCD manages
* Good, because teardown is decoupled; adding new components requires no teardown changes
* Good, because the ArgoCD instance remains auto-synced from the repo
* Bad, because `config.json` needs a `standalone` flag not present in other components
