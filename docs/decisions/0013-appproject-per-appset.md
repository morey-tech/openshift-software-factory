# Separate AppProjects for Operators and Operands ApplicationSets

## Context and Problem Statement

Argo CD ships with a single `default` AppProject that permits any source repository, any destination cluster and namespace, and any resource kind. The two ApplicationSets (`operators` and `operands`) were initially configured to use `default`. Should dedicated AppProjects be created, and if so, at what granularity?

## Decision Drivers

* Principle of least privilege: Applications should only be able to deploy resources they actually need
* Source integrity: only manifests from this repository should be deployable through these ApplicationSets
* Auditability: separating operators from operands makes it easier to reason about what each set of Applications is allowed to do at the cluster level
* Operators install cluster-scoped resources (ClusterRole, ClusterRoleBinding, ConsolePlugin) that operands do not need

## Considered Options

* **Keep `default` project** â€” no change; all Applications share the unrestricted default project
* **One AppProject for all Applications** â€” single project replacing `default`, with source and destination restrictions
* **Separate AppProject per ApplicationSet** â€” `operators` project for the operators AppSet, `operands` project for the operands AppSet

## Decision Outcome

Chosen option: **Separate AppProject per ApplicationSet**, because it allows the cluster-scoped resource whitelist to be scoped to only what each AppSet actually needs â€” operators require `ClusterRole`, `ClusterRoleBinding`, and `ConsolePlugin` while operands only need `Namespace` (for `CreateNamespace=true`).

### Consequences

* Good, because operand Applications cannot accidentally (or maliciously) create cluster-scoped RBAC resources â€” only the operators project permits those kinds
* Good, because both projects restrict `sourceRepos` to this repository, preventing rogue Applications from deploying arbitrary content
* Good, because both projects restrict `destinations` to the local cluster, preventing cross-cluster deployments
* Good, because the separation creates a natural place to add per-project sync windows, notifications, or roles in the future
* Bad, because the `operators` cluster resource whitelist must be kept up to date as new components are added â€” a component that creates a new cluster-scoped resource kind will fail to sync until the whitelist is updated
* Bad, because two objects to maintain instead of one

### Confirmation

Verify with:
```bash
oc get appproject -n openshift-gitops
```
Both `operators` and `operands` projects should exist. Existing Applications generated by the AppSets should show their project in:
```bash
oc get application -n openshift-gitops -o custom-columns='NAME:.metadata.name,PROJECT:.spec.project'
```

## Pros and Cons of the Options

### Keep `default` project

* Good, because zero configuration required
* Bad, because any Application can deploy any resource kind to any cluster from any repository
* Bad, because there is no audit boundary between operators and operands

### One AppProject for all Applications

* Good, because source and destination restrictions apply to all Applications
* Bad, because the cluster resource whitelist must include the union of all resource kinds needed by both operators and operands, making it harder to reason about what each group is actually allowed to do

### Separate AppProject per ApplicationSet

* Good, because cluster resource whitelists are minimal and purpose-specific
* Good, because each project's allowed kinds directly document what that class of Applications deploys
* Neutral, because namespace resource whitelist is `*/*` in both projects â€” namespace-level resources are intentionally unrestricted since the source repo restriction already gates what can be deployed
